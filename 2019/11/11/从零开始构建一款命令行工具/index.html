从零开始构建一款命令行工具

发表于2019-11-11|更新于2021-05-20|前端基础建设
|阅读量:

拥有一个好的命令行工具一定是事半功倍的，它既可以快速构建一个新的工程，也可以生成页面、项目打包、快速查看文档、启动一个服务，甚至与服务器通信，完成很多自动化批处理高耗时任务，可以说，只有你想不到，没有它做不到。

什么是命令行工具

引用维基百科的一个官方解释：命令行界面（英语：Command-Line Interface，缩写：CLI）是在图形用户界面得到普及之前使用最为广泛的用户界面，它通常不支持鼠标，用户通过键盘输入指令，计算机接收到指令后，予以执行。也有人称之为字符用户界面（character user interface, CUI）。

通俗一点说就是，执行一个脚本命令，完成一个或者一系列的操作。

为什么要做命令行工具

  • 提升效率

试想一下，如果我们需要频繁的创建新的项目，在开发中，又需要高频率的新建页面和组件，在没有 CLI 的时候，完全就是复制以前的项目代码进行修改，删除之前项目的业务部分，如果项目类型不一样，很可能还要修改配置，开发时，想要新建一个页面，也是复制粘贴来改，这种重复的事情多了之后，自然就影响开发效率了，如果有 CLI 的存在，新建项目只需要执行一个命令就可以完成，新建页面也是如此，那么效率自然能得到很大提升，其次，如果 CLI 本身附带了很强的业务属性，例如批量的文件上传，那效率的提升更是显著。

  • 最佳实践的沉淀

对于一个团队来说，在做好业务的同时，也会对外输出技术解决方案，而前端的基础建设，它不是一蹴而就的，而是一个积累的过程。在建设 CLI 的过程中，团队的每个成员都能够参与进来，例如提供模板脚手架，以及对模板脚手架进行不断迭代优化等，随着业务的不断变化，沉淀的技术不断完善，最终形成某个行业的解决方案，那么最终得到的就是最佳实践的沉淀。

选择辅助工具

我们前面的工作已经配置好了一个命令，那么接下来就是正式的开发阶段了。基本一个命令都会有如下的一些参数：

  • -h, –help：查看版本号
  • -V, –version：查看帮助

而每个命令又会有自定义的参数，这样一来，如果每个命令的参数都需要要自己来解析，那么会变得非常麻烦，所以我们需要选一个适合自己的命令行辅助工具。以下是几个比较常用的：

  • commander
  • caporal
  • cac
  • optionator

发布

开发完成后，接下来就是发布到 npm 上接受社区的毒打了。

1、注册一个 npm 账号。

2、如果是第一次发布，需要使用 npm adduser 来添加账号、密码和邮箱。

3、执行 npm publish 发布。

发布的时候建议使用作用域包，就是使用艾特符号（@）开头来命名包名，以解决包同名问题，如果是第一次发布作用域包，会报错：

这时候需要指定 –access 参数， 像这样：npm publish --access public。

有时候 npm registry 指定得不对也会出现这个错误，推荐使用 nrm 来管理 npm registry。

更进一步

以上只是一个从零开始创建一个 CLI 的一小步，而一个真正好用的 CLI，还应该有以下几个点：

  • 完整的命令行功能，支持 关键字、**–、-** 传参语法。
  • 默认配置，就像 webpack 一样，有默认配置，也可以自己设置值。
  • 可扩展性，支持添加插件。
  • 更加友好的体验，例如等待动画。

而一个真正好用的 CLI，绝不是仅限以上几点，它一定是像一款产品一样，需要经过不断的完善，结合自身业务发挥出更大的优势。

常用的命令行相关工具库
